● The problem

AI agents need specs to be safe

Greenfield: write spec, agent executes. Legacy or vibe-coded: no spec → agent has no idea what it cannot break.

WITHOUT AEGIS vibe-driven evolution
  • 10-year-old logic, no documentation
  • Implicit business rules nobody can cite
  • Agent rewrites a function, breaks an invariant
  • Drift between code and Confluence wiki
  • "Why did it work yesterday?" loop
  • Refactor blocked by fear of unknown contracts
WITH AEGIS spec-as-contract
  • Specs extracted from real code, line-traceable
  • Contracts marked 🟢 confirmed / 🟡 inferred / 🔴 gap
  • Drift loop catches every diverging edit
  • Policy gate blocks signature breaks at the PR
  • Knowledge graph shows blast radius before merge
  • Agents read specs first, then write

● Pipeline

4 stages — Specify, Understand, Guard, Deploy

Discovery extracts specs. Migration plans the rewrite. Build delegates to your coding agent. Control plane stops drift.

STAGE 1
Discovery
Scout · Archaeologist · Detective · Architect · Writer · Reviewer → legacy code into aegis/ specs.
STAGE 2 (opt)
Migration
Paradigm Advisor · Curator · Strategist · Designer · Inspector → migration plan + parity tests.
STAGE 3
Build
Hand off to your coding agent (Claude · Codex · Cursor · Gemini · Kimi) — the migration plan drives new code.
STAGE 4 ⭐
Control plane (this fork)
Keeper + Graph + Policy gate (file-level + signature diff). Drift never lands.

● Trust model

Every claim labelled with confidence

Hallucination is the mortal enemy of legacy specs. Aegis annotates every statement so humans know what to trust.

🟢

CONFIRMED

Extracted directly from code. Citable with file path and line number. Safe to cite in PRs.

e.g. src/auth/login.js:42
🟡

INFERRED

Deduced from patterns or naming. May be wrong. Validate with stakeholder before relying on it.

e.g. naming convention says retry, no test confirms
🔴

GAP

Not determinable from code alone. Lands in aegis/reports/questions.md for human resolution.

e.g. business policy not in code
